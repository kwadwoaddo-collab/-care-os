# Operations Control Center — Architecture

## Overview

The Operations Control Center (OCC) transforms Care OS from a compliance & intelligence
platform into a real-time care operations coordination system. It gives coordinators,
managers, and compliance leads a single command center to manage daily operational
pressure, track unresolved issues, coordinate handovers, and run safe daily care
operations.

---

## System Architecture

### Layer Stack

```
Browser
  └── OCC Page (/admin/operations)            — Server component
        └── OperationsControlCenter.tsx        — Client component (interactive)
              ├── PriorityQueuePanel             — reads from OccSummary
              ├── LiveFeedPanel                  — reads from OccSummary
              ├── ShiftCoveragePanel             — reads from OccSummary
              ├── SafeguardingPanel              — reads from OccSummary
              ├── HandoverPreview                — reads from OccSummary
              └── ComplianceAlertsPanel          — reads from OccSummary

API Layer
  ├── GET /api/admin/operations/summary         — Main OCC payload (all in one)
  ├── GET /api/admin/operations/queue           — Paginated queue list
  ├── POST /api/admin/operations/queue          — Create queue item
  ├── PATCH /api/admin/operations/queue/[id]   — Update/resolve queue item
  ├── GET /api/admin/operations/handover        — List handover notes
  ├── POST /api/admin/operations/handover       — Create handover note
  ├── GET /api/admin/operations/briefing        — Daily briefing payload
  └── GET /api/admin/operations/feed            — Chronological event feed

Database
  ├── operations_queue        — Persistent operational action items
  ├── handover_notes          — Shift-to-shift handover records
  └── (existing tables)       — incidents, shifts, documents, compliance_overrides,
                                 staff_profiles, audit_logs
```

## Data Model

### `operations_queue`

Persistent queue of operational action items. Can be manually created by coordinators
or auto-generated by system sweeps (future).

| Column | Type | Description |
|--------|------|-------------|
| id | UUID | PK |
| company_id | UUID | Tenant isolation |
| priority | TEXT | critical / urgent / warning / informational |
| category | TEXT | safeguarding / compliance / staffing / onboarding / incident / medication / shift_coverage / other |
| title | TEXT | Action item title |
| description | TEXT | Optional detail |
| entity_type | TEXT | Linked entity type (staff, client, shift, incident) |
| entity_id | UUID | Linked entity ID |
| entity_url | TEXT | Direct link to entity |
| assigned_to | TEXT | Assignee name/email |
| assigned_at | TIMESTAMPTZ | When assigned |
| due_date | TIMESTAMPTZ | When action is due |
| status | TEXT | open / in_progress / resolved / dismissed |
| auto_generated | BOOLEAN | System vs manual |
| source | TEXT | What generated it |
| escalation_triggered_at | TIMESTAMPTZ | When escalated |
| escalation_acknowledged_at | TIMESTAMPTZ | When acknowledged |
| escalation_acknowledged_by | TEXT | Who acknowledged |
| resolution_notes | TEXT | How it was resolved |
| resolved_at | TIMESTAMPTZ | When resolved |
| resolved_by | TEXT | Who resolved it |

### `handover_notes`

Shift-to-shift coordination records.

| Column | Type | Description |
|--------|------|-------------|
| id | UUID | PK |
| company_id | UUID | Tenant isolation |
| handover_date | DATE | Date of handover |
| shift_period | TEXT | morning / afternoon / evening / night / day |
| author_name | TEXT | Coordinator who wrote the note |
| author_id | UUID | Linked profile (optional) |
| summary | TEXT | Main handover text |
| flagged_items | JSONB | Array of {type, description, priority, entity_id?} |
| follow_up_actions | JSONB | Array of {action, owner?, due?, done?} |
| status | TEXT | active / reviewed / archived |
| reviewed_by | TEXT | Who reviewed |
| reviewed_at | TIMESTAMPTZ | When reviewed |

## Feature Modules

### 1. Operations Control Center Dashboard (`/admin/operations`)

Single-page operational command center. Loads from `/api/admin/operations/summary`
which runs 8 parallel Supabase queries:

- Open incidents (for counts, safeguarding queue, feed)
- Shifts next 7 days (uncovered count, 24h coverage)
- Onboarding stalls (staff in pre_employment >14 days)
- Critical documents expiring (DBS, RTW in next 30 days)
- Active compliance overrides
- Operations queue top items
- Latest handover note
- Recent incidents for feed

**No additional round trips from the browser** — all data is server-rendered.

### 2. Operational Priority Queue (`/admin/operations/queue`)

Interactive client-side queue manager.

- Filtered by priority, category, status
- Inline expand for full details
- Mark in progress / resolve with notes / dismiss
- Create new items with full form
- Resolving triggers PATCH to API, updates local state optimistically

### 3. Handover System (`/admin/operations/handover`)

Shift-to-shift coordination notes.

- Structured creation: author, period, date, summary
- Flagged items list (type + description + priority)
- Follow-up actions list (action + owner + due date)
- Historical view with expand/collapse
- Status tracking (active → reviewed → archived)

### 4. Daily Operational Briefing (`/admin/operations/briefing`)

Generated operational brief covering:

1. **Safeguarding** — unresolved incidents, severity breakdown
2. **Incident Risk** — high/critical incidents last 7 days, medication errors
3. **Staffing Pressure** — uncovered shifts next 7 days
4. **Compliance Deterioration** — DBS / RTW expiring or expired
5. **Onboarding Bottlenecks** — staff stuck >14 days in pre_employment
6. **Active Overrides** — compliance bypass summary

Each section has `clear / warning / critical` status. The page shows a top-level
risk headline based on worst-case section status.

### 5. Live Operational Feed (`/api/admin/operations/feed`)

Chronological event stream from:
- Recent incidents (with severity and linked client)
- Handover notes created
- Queue items created/resolved
- Audit log events (override granted/revoked)

### 6. Shift Coordination Intelligence (embedded in OCC)

The OCC summary includes 24h shift coverage:
- Total shifts in next 24h
- Covered vs uncovered count
- Specific uncovered shift details with times and client names
- Coverage percentage with colour-coded bar

## API Reference

### `GET /api/admin/operations/summary`
Returns `OccSummary` — full OCC payload. No query params.

### `GET /api/admin/operations/queue`
| Param | Type | Description |
|-------|------|-------------|
| priority | string | Filter by priority |
| category | string | Filter by category |
| status | string | Filter by status (default: open) |
| page | number | Page (default: 1) |
| pageSize | number | Per page, max 50 (default: 25) |

### `POST /api/admin/operations/queue`
Body: `{ title, priority, category, description?, assigned_to?, due_date? }`

### `PATCH /api/admin/operations/queue/[id]`
Body: any subset of `{ priority, category, title, description, assigned_to, due_date, status, escalation_acknowledged_by, resolution_notes, resolved_by }`

Setting `status: 'resolved'` auto-sets `resolved_at`. Setting `assigned_to` auto-sets
`assigned_at`. Setting `escalation_acknowledged_by` auto-sets `escalation_acknowledged_at`.

### `GET /api/admin/operations/handover`
| Param | Type | Description |
|-------|------|-------------|
| limit | number | Max results, max 20 (default: 10) |
| date_from | string | ISO date filter |

### `POST /api/admin/operations/handover`
Body: `{ author_name, summary, shift_period?, handover_date?, flagged_items?, follow_up_actions? }`

### `GET /api/admin/operations/briefing`
Returns `DailyBriefing` — no query params. Always fresh.

### `GET /api/admin/operations/feed`
| Param | Type | Description |
|-------|------|-------------|
| limit | number | Max events, max 50 (default: 30) |
| since | string | ISO timestamp (default: 7 days ago) |

## Preserved Systems

All existing systems are unchanged:

- **Incident Intelligence** (`/admin/incidents/intelligence`) — risk engine, staff/client risk profiles
- **Compliance Intelligence** (`/admin/compliance`) — document tracking, risk scores, escalation
- **Scheduling Intelligence** (`/admin/shifts`) — workforce capacity, shift recommendations
- **Workforce Intelligence** (`/admin/workforce`) — capacity planning, operational readiness
- **RBAC** — all permissions preserved, Operations uses existing `incidents:read/write`
- **Audit logging** — queue item updates are logged to `audit_logs`
- **Multi-tenant isolation** — all queries filter by `company_id`
